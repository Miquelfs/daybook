## Part 1 — Where the project actually stands

### 1.1 Shipped and working (far beyond what CHANGELOG/VISION admit)

The June–July build-out delivered, on top of Phases 1–3:

- **Money**: full investor portfolio (holdings, yfinance `price_sync.py`, snapshots, DCA `investment_plans` + `plan_executor.py`, holding detail pages), account/category drill-downs, anomaly Z-scores, forecast, budget-velocity alerts (`/money/overview`).
- **Groceries**: Mercadona client, pantry, price history + comparison, receipt OCR round-trip (iOS Vision → `/groceries/receipts/parse`), LLM meal planner. Full web UI under `/money/groceries`.
- **Training**: the **Omyra coach** (adaptation engine, 12 plan templates, race goals, RPE loop, taper phases, compliance), readiness score, CTL/ATL/TSB, per-activity metrics (NP/IF/VI/EF/decoupling), injuries flow (table + API + web + iOS).
- **Health**: sleep section (`/health/sleep`, summary/stages/correlations), Load Index (`/health/load`, `compute_load_index.py` — Horizon 1 embryo is real), streaks, intraday HR.
- **Aviation**: duty roster import + day briefs + pay estimate, aircraft/airport/captain drill-downs, night-seconds backfill.
- **Horizon 3 instruments**: decision log with calibration (`/decisions`), N-of-1 experiments with Cohen's d (`/explore/experiments`).
- **AI layer**: morning brief, health narratives (4 topics, 24h cache), weekly expense summary — all coded with graceful degradation.
- **iOS app**: complete 5-tab client (~16.9k LOC, zero third-party deps) covering nearly every read surface + write flows (questionnaire w/ offline queue, flights, expenses, books, injuries, decisions, experiments, roster). Plus native-only: receipt OCR, FinanceKit import, background GPS→Overland, 3 widgets, share-extension expense capture.

### 1.2 Critical risks found (fix before building anything new)

1. **~176 files uncommitted across the two repos.** daybook: 65 modified + 111 untracked (~8.4k insertions) — the entire recent build-out. daybook-ios: one initial commit; effectively the whole app is unversioned. One bad `git clean`/disk event loses a month of work. → checkpoint commits now.
2. **All AI features are dark.** The HP Pavilion Ollama node was deprecated 2026-06-29; `.env` no longer defines `OLLAMA_HOST` while `ollama_client.py`, `daily_sync.sh`, morning brief, narratives, receipt parsing and meal planner still depend on it. Everything degrades gracefully but silently. → decision needed (see 4.1-c).
3. **Backups don't leave the Pi.** `backup.sh` gzips DBs to `data/backups/` **on the same SD card**, and the Pi holds the only copy of moods, photos, money. → off-device backup is the single highest-leverage reliability fix.
4. **Docs lie about the present.** CHANGELOG ends 2026-05-18; VISION still says "Phase 4 iOS: NEXT" (it shipped); the PWA-hardening doc contradicts the shipped native app; no ADR records the iOS-vs-PWA decision. Zero tests exist in either repo.
5. **Morning brief runs hourly**, not at 6am — `cron-install` schedules `daily_sync.sh` at `0 * * * *` and the brief is its last step, so it regenerates all day.

### 1.3 Strategic tension to close (recommended: ADR-004)

VISION says PWA/web; the codebase says native iOS + web. Reality is good: **web = deep-dive/desktop instrument, iOS = daily capture + glanceable client, Pi API = single contract.** Write that down as ADR-004 so future work stops re-litigating it.

## Part 3 — Parity: state and the mechanism to keep it

### 3.1 Current gaps

**iOS lacks (web has):** portfolio + holding detail; money category/account drill-downs; `/money/insights` (once built); groceries meal-plan + price charts; dedicated sleep / load / streaks views; aviation aircraft/airport/captain drill-downs; place detail (`TODO` already in code); databases browser (fine to stay web-only).

**Web lacks (iOS has):** receipt scanning (needs camera — fine), FinanceKit (platform-bound — fine), quick-add sheet ergonomics, widgets/glanceability. Web equivalent = better PWA installability + a compact "capture bar."

### 3.2 The mechanism (architecture improvement)

Parity today is manual triple-entry: Pydantic → `lib/api.ts` → Swift structs with hand-written CodingKeys. Two structural fixes:

1. **Generate types from the API.** FastAPI already emits OpenAPI. Add `make types`: `openapi-typescript` → `lib/api-types.gen.ts`; Swift OpenAPI Generator (or a small Python script emitting structs with explicit snake_case CodingKeys, matching the existing pattern) → `Models/Generated.swift`. Hand-written types shrink to view-models only.
2. **Parity checklist as code.** A tiny script (`scripts/parity_report.py`) diffing: API routes ⟷ web proxy routes ⟷ iOS `APIClient` paths, emitting a markdown table. Run it in `make verify`. Parity drift becomes visible instead of remembered.

Also fix on iOS: JSONEncoder gets `.convertToSnakeCase` (kills most hand-written CodingKeys); route photo upload through `DaybookAPIClient`; extend `OfflineQueue` beyond the questionnaire to all POST/PATCH writes (expenses and flights logged in airplane mode is the whole point of a pilot's logbook app).

## Part 4 — Improvements

### 4.1 Architecture & ops
- **a. Version control discipline**: checkpoint commit now; then small thematic commits; update CHANGELOG per sprint.
- **b. Migrations**: consolidate ad-hoc `migrate_*.py` into numbered idempotent migrations (`infrastructure/db/migrations/NNN_*.{sql,py}`) + a `migrate.py` runner tracking `schema_version`; keep base `schema.sql` current so fresh installs work.
- **c. LLM backend**: keep the `ollama_client.py` abstraction; point `OLLAMA_HOST` at Ollama on the Mac via Tailscale (zero code change, free, local-first) — accepts that briefs only generate when the Mac is awake, which the graceful-degradation design already tolerates. Optional later: pluggable cloud provider for narratives. AI stays a lens, never a gate (VISION §6).
- **d. Backups off the Pi**: nightly Mac-side `launchd` pull job — `rsync pi:~/daybook/data/backups/ + data/photos/` to the Mac + external drive. Never Mac→Pi (CLAUDE.md rule stands).
- **e. Cron split**: hourly `daily_sync.sh` for data; morning brief guarded to generate once/day in the 06:00 window (or a separate 6am cron line).
- **f. Tests where money/math lives**: a thin pytest layer for money sign/velocity math, correlation stats, load index, Omyra adaptation rules — the four places a silent regression actually hurts. No UI test theater.
- **g. Security**: API is open on Tailscale (acceptable single-user posture); keep Overland bearer; consider a static bearer for mutating endpoints when convenient. Low priority.
- **h. ADR-004**: record the two-client strategy (web=instrument, iOS=capture, API=contract).

### 4.2 Design
- **Web IA restructure**: 40+ pages have outgrown the flat nav. Proposal — five hubs mirroring the iOS tabs: **Today · Life (journal/moments/timeline/life/decisions) · Body (health/sleep/load/training/stats/injuries) · Money (incl. groceries/portfolio/insights) · World (explore/places/trips/aviation)**. Same mental model on both clients.
- **Shared design tokens**: one `tokens.json` (colors incl. sport palette run=coral/bike=blue/swim=teal, life-event colors, spacing, type scale) → CSS variables + a generated `DesignSystem.gen.swift` section. iOS `DesignSystem.swift` already mirrors web colors by hand; make it mechanical.
- **Dataviz consistency**: standardize chart idioms (same zone colors, same sparkline style) across Recharts-style web charts and the hand-drawn SwiftUI charts; consider Swift Charts adoption where it reduces code.
- **Empty/dark states**: every AI-dependent card should say *why* it's empty ("LLM node offline") instead of vanishing — honesty beats mystery.

### 4.3 Flows (how the app is used)
- **Evening ritual** is strong (30-second questionnaire, offline). Add: one-tap RPE prompt when the day has a training session (feeds Omyra), decision-review prompt when a decision hits its horizon date.
- **Morning ritual**: 6am brief + "what's coming" (roster day brief + planned session + weather + streaks) — the Today view splits *what happened / what's coming* (Horizon 4 seed).
- **Weekly review (the missing ritual)**: Sunday auto-page — did I do what I said (plan-vs-actual), decisions made + predictions, experiments status, spend vs budget, *did I use Daybook this week?* NORTH_STAR names this the supreme question; make the system ask it.
- **Capture friction**: iOS quick-add is good; add App Intents ("log expense", "log mood") for Siri/Action Button; web gets a global "+" capture bar.
- **Notifications**: self-hosted **ntfy** on the Pi (boring-tech-compliant) for: questionnaire reminder, brief ready, anomaly alerts, decision due, budget velocity breach. This is what turns Daybook from a place you go into a system that reaches you.

## Part 5 — Dream big (use-case notes, Pi-test compliant)

- **Layover copilot** (roster × GPS × restaurants × health × memory): land in a city → Daybook surfaces past visits, where you ate, what you felt, sleep strategy for the next duty. The aviation-native killer feature no product can build.
- **Load model that predicts** (Horizon 1 finish): validate Load Index against next-day energy/mood (lag-1/lag-2); then run it **forward** over the imported roster: "this earlies-block will cost you ~15 HRV points" — before the month starts. Same engine answers "can I absorb this training week?" (Ironman convergence).
- **Memory machine** (Horizon 2): sqlite-vec embeddings over notes/answers/decisions → "days like today", "when did I last feel like this?"; layover audio diary via Whisper — 30s of voice becomes searchable life-corpus. Corpus density is the moat.
- **Daybook MCP server**: expose read-only tools (`get_day`, `search_days`, `correlate`, `spend_summary`) over the existing API so Claude/any LLM client can converse with your life. NL interface without hosting a model.
- **Experiments engine v2**: correlation engine auto-*suggests* protocols ("alcohol on layovers ↔ −12 HRV — run the 3-week experiment?") — closing the record→model→instrument ladder.
- **Year in Review artifact**: generated January PDF/page — the annual proof-of-life; plus yearly Parquet cold export (20-year principle).
- **Personal inflation index** from receipt line-items across supermarkets (data already flowing).
- **Calibration curve**: after ~50 resolved decisions, plot stated confidence vs outcomes — "your 80% is a 62%."
- **E-ink day display** (pi-reader synergy): today's brief + streaks on the wall.
- **Anticipatory Today**: the view's top half becomes tomorrow-aware (roster brief, planned session, weather, predicted load).
